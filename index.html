方案一：切换到旧版输入法

这是最快捷的临时解决方案，能让输入法立刻恢复正常，操作步骤为：

  • 在任务栏右下角右键点击“中/英”输入法图标，选择设置
  • 在设置窗口中点击常规
  • 找到兼容性部分，打开使用以前版本的微软拼音输入法开关。

方案二：安装缺失的依赖库（更彻底）

这是从根源上解决问题的方法，并且已经被广泛验证有效。其步骤为：

  1. 下载文件：依赖库文件Microsoft.VCLibs https://dl.lancdn.com/landian/soft/vclibs
  2. 解压文件：将下载的压缩包解压，你会得到两个.Appx格式的文件。
  3. 放置文件：将这两个.Appx文件复制到C:盘的目录下。
  4. 安装文件：
    • 右键点击开始菜单，选择Windows PowerShell(管理员)或终端(管理员)
    • 在打开的窗口中，根据你的系统类型，输入以下命令并回车
      • 64位系统
      • 32位系统
  5. 重启电脑：命令执行成功后，重启电脑即可生效。

[!WARNING] 如果提示“部署失败”或“资源正在使用中”，可以重启电脑后再试一次。

提取Word中的高清图片

发表于 2026-06-22

直接复制Word中的图片可能会因为剪贴板格式转换或Word默认的压缩机制导致画质下降和背景丢失。

docx 文档提取高清图片

为了提取无损且高清的原图，可以将.docx文件修改为.zip后解压。这种方法能够确保图像不失真。这是因为现代的 .docx 文件本质上是一个包含多种资源（包括原始图片）的压缩包。通过解压，可以直接获取未经任何处理的原始高清图片。

操作步骤：

  • 备份文档：建议先复制一份需要提取图片的Word文档。
  • 更改后缀名：将文档的后缀名从 .docx 改为 .zip（如果看不到后缀名，需在文件夹选项中开启“显示文件扩展名”）。
  • 解压文件：使用解压缩软件（如7-Zip、WinRAR等）打开或解压该ZIP文件。
  • 提取图片：在解压出的文件夹中，依次进入 word → media 目录。这里存放了文档中所有的原始图片，您可以直接将它们复制出来。
  • 注：此方法仅适用于 .docx 格式，不适用于老旧的 .doc 格式。

doc 文档提取高清图片

由于老旧的 .doc 格式并非基于 XML 的压缩包结构，因此无法直接使用“修改后缀为 ZIP”的方法。一种最简便的方法是使用MsOffice将.doc文档转换为.docx文档，然后再修改后缀为.zip, 使用解压缩软件解压缩。另一种方法是另存为网页，这是最古老且最可靠的方法，能够很好的保留图片的原始画质。

操作步骤：

  • 用 Word 打开需要提取图片的 .doc 文档。
  • 点击左上角的“文件” → “另存为”。
  • 选择好保存位置，在 “保存类型” 下拉菜单中，选择 “网页 (*.htm; *.html)”（注意：请勿选择 “单个文件网页”），然后点击保存。
  • 保存后，系统会在同目录下自动生成一个名为 “[文档名]_files” 的文件夹。打开该文件夹，里面就整齐排列着文档中的所有原始高清图片。

这个错误信息的意思是，Lua 语言服务器（lua_ls）被你的编辑器（如 VS Code 或 Neovim）告知，要将整个 /home/foobar 目录作为工作区进行扫描，而服务器由于性能或安全方面的考量，拒绝了这个指令。这通常是一个配置问题，可以通过正确设置工作区解决。

解决方案

在你的Neovim配置目录下创建.luarc.json文件，明确告诉lua_ls这就是项目根据目录。即

其中：runtime.version一行，指明了兼容Lua 5.1版本。同时使用workspace.ignoreDir来忽略无关目录，以提高性能，所以需要配置这个忽略列表。

为什么之前会报错？

  • 默认情况下，nvim-lspconfig 的 lua_ls 如果没有找到项目根目录，会回退到使用当前工作目录（vim.loop.cwd()）。如果你是在主目录下启动 Neovim（例如直接运行 nvim ~/.config/nvim/init.lua），那么当前工作目录就是 /home/foobar，导致 lua_ls 被设置为扫描整个主目录，从而触发拒绝。

  • 创建 .luarc.json 后，root_pattern 会优先匹配，从而正确设置工作区。

  • 按上述步骤操作后，错误信息就会消失，LSP 功能也能正常使用。

Firefox浏览器打开网页遇到HSTS错误的解决方法

发表于 2026-06-11

今天打开msn的网页时，遇到了问题：

[!WARNING] www.msn.cn 启用了被称为 HTTP 严格传输安全（HSTS）的安全策略，Firefox 只能与其建立安全连接。您无法为此网站添加例外，以访问此网站。

这个问题的根源在于网站启用了一项名为 HTTP 严格传输安全 (HSTS) 的安全策略。这本身是一种网站保护用户安全的机制，可以强制浏览器始终使用加密的 HTTPS 连接。简单来说，主要原因是 www.msn.cn 很可能被加入了各大浏览器（包括 Firefox）共同维护的“HSTS 预加载列表”中。这个列表被硬编码在浏览器里，针对列表中的网站，浏览器会强制执行 HTTPS 连接，并且不允许用户添加例外，所以您无法通过常规方式访问。要解决这个问题，您可以尝试以下几个方法

在 Firefox 中清除网站HSTS记录

如果您的电脑本地并没有访问过 www.msn.cn 的历史记录，可以尝试通过“忘记此网站”功能来清除可能存在的 HSTS 策略：

  1. 按下 Ctrl + H 打开 Firefox 浏览器中的历史记录侧边栏。
  2. 在侧边栏的搜索框中输入 msn.cn 查找相关记录。
  3. 右键点击搜索结果并选择“忘记此网站”。
  4. 在弹出的确认框中再次点击“忘记”。

[!WARNING] 此操作会清除 Firefox 本地存储的关于该网站的所有信息，包括强制的 HSTS 规则。

修改高级配置

  1. 在 Firefox 地址栏输入 about:config 并回车。
  2. 点击“接受风险并继续”按钮。
  3. 在搜索框中输入 network.stricttransportsecurity.preloadlist。
  4. 双击找到的首选项，将其值从 true 改为 false。
  5. 修改后，无需重启 Firefox，即可再次尝试访问 msn.cn。

[!WARNING] 如果以上方法都无法解决问题，您可以通过修改 Firefox 的高级配置来强制禁用整个 HSTS 预加载列表。请注意，这会显著降低您浏览器访问其他网站的安全性，请您谨慎操作并自行承担风险。

Linux目录双向同步完全指南：Unison从入门到实战

发表于 2026-04-28

在日常工作和运维中，我们经常遇到这样的需求：两个目录 A 和 B 都可能随时新增或修改文件，希望它们最终保持完全一致，同时绝不能删除任何文件。这种场景下，传统的 rsync 单向同步方案显得力不从心，而 Unison 正是解决这一问题的利器。

本文将深入讲解 Unison 的安装、使用、冲突处理机制，以及如何安全地实现双向同步。

一、为什么 rsync 不够用？

很多人首先想到的是 rsync，但它的局限很明显：

问题所在

  • rsync 本质是单向 工具，只能从源到目标。
  • --update 只保护目标端的新文件，不会反向同步。
  • 即使执行两次rsync， 也无法安全处理两边同时新增或修改的场景。

结论 纯新增文件（从不修改）的场景可以用两次rsync拼凑，但是只要涉及修改，就需要真正的双向同步工具。

二、Unison 简介

Unison 是一款跨平台的双向文件同步工具，核心特性：

  • 真正的双向同步: 自动检测并传播两边的变更
  • 安全第一： 冲突时不会自动覆盖，需要明确处理
  • 增量传输：只同步差异部分，节省带宽和时间
  • 跨平台：Linux、macOS和Windows均可使用
  • 支持SSH: 可同步远程服务器目录

基础用法

最简单的双向同步

执行后，Unison会扫描两个目录，并逐一询问如何处理每个差异。

自动模式（跳过非冲突确认）

  • 作用：自动接受所有非冲突的变更（新增、更新、删除）
  • 冲突：仍然会停下等待用户选择

完全静默模式（适合脚本/写明任务）

  • 作用：不问任何问题
  • 冲突时：直接跳过，不做任何处理
  • 注意：冲突会导致目录不一致，需配合-prefer newer使用

-batch 直接跳过，不处理 不适合，除非配合-prefer
-batch -prefer newer 自动选择修改时间更新的版本 日志、缓存等可自动决策的数据
